# ADR-0006:门禁布局:Stop 硬门禁 + PostToolUse 非阻断反馈

状态:Accepted

## 背景

门禁有两个设计变量:放在哪个 hook、是否阻断。放错位置或全部阻断,会把质量保证变成开发阻力;全不阻断,无人值守就失去「完成即可信」的保证。

## 备选与否决理由

- **PostToolUse 阻断式(每次编辑必须 tsc/lint 全绿才继续)**:看似最严格。否决:重构过程中的中间态报错是合法状态(先改类型定义、后改调用方),强制即时修复会造成修复抖动:AI 被迫在半成品上打补丁,补丁又成为下一轮报错源。
- **仅 CI 收口(本地无门禁)**:常规团队做法。否决:反馈延迟到 push 之后,届时修改意图已离开上下文窗口,修复成本最高。
- **Stop 硬门禁 + PostToolUse 非阻断**:采纳。

## 决议

- `PostToolUse(Edit|Write)`:对改动文件跑 `tsc --noEmit` + ESLint,结果作为**非阻断信息**回灌。AI 随写随知,但保有「先完成结构调整、再统一修复」的自由。
- `Stop`:AI 宣称任务完成时,强制执行 `pnpm gate`(typecheck + lint + test + build)。红灯则 block 停止,失败输出回灌,继续修复。绿灯是「完成」的唯一定义。
- 门禁命令唯一化:hook 与人跑的都是 `pnpm gate`,门禁定义只存在一处。

## 后果

- 每个任务收尾增加 20-40s 全量门禁耗时。取舍立场:无人值守场景下时间不稀缺,可信的完成状态才稀缺
- 「红灯时修改裁判」成为新的逃生通道,由 ADR-0008 封堵

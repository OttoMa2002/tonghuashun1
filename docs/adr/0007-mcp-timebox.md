# ADR-0007:MCP 渲染验证设 2 小时时间盒,超时降级为文档 + 人工留证

状态:Accepted

## 背景

单测有结构性盲区:能验证降采样算法正确、协议类型正确,无法验证「canvas 真的画出了东西」和「百万点渲染耗时是否在预算内」。而这恰是本项目最核心的交付物。chrome-devtools MCP(浏览器自动化)能闭合这个环:启动 dev server → 打开 million-points 页 → 截图断言非空白 → 读取渲染耗时。但其配置成本存在不确定性,不可让单个验证手段吞掉项目工期。

## 备选与否决理由

- **不做视觉/性能验证**:零成本。否决:自检闭环在最核心的考点上开口,「AI 生成代码后的稳定性」失去渲染侧证据。
- **不设上限投入直至跑通**:验证最完整。否决:范围失控风险;验证手段的成本不应超过被验证物。
- **时间盒 + 降级路径**:采纳。

## 决议

MCP 渲染验证的搭建投入上限 **2 小时**:

- 时间盒内跑通:进入自检闭环,渲染类任务的 DoD 增加 MCP 验证项
- 超时:降级为两件套——(1)验证方案设计文档(工具选择、断言项、性能预算指标);(2)人工验证留证:截图与渲染耗时记录提交至 repo,作为 commit 证据链一部分

降级不影响其余门禁(Stop 全量门禁照常)。

## 后果

- 两种结局都有可提交的交付物,无沉没成本风险
- 渲染耗时预算需在 data-contract.md 或任务卡片中先行定义(如 raw 百万点首帧 < 1s),否则验证无断言对象
- 该取舍本身展示了对验证手段的成本意识:验证体系也要做工程取舍,不是越多越好

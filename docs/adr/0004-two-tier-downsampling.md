# ADR-0004:两级降采样(step + LTTB),raw 为受控例外

状态:Accepted

## 背景

矛盾:题目要求展示百万级数据渲染能力,但日常 dashboard 若总是传输和处理原始精度数据,性能与带宽都是浪费。降采样策略必须同时回应两端。

## 备选与否决理由

- **全部走 raw**:渲染能力展示最充分。否决:dashboard 常规视图传输百万点是无意义负载,也违背真实监控产品的做法。
- **全部走降采样**:性能最稳。否决:永远渲染不到百万点,题目核心考点落空。
- **仅客户端单级 LTTB**:实现最简单。否决:原始数据全量传输到 Worker 再压缩,传输浪费;且失去 step 这个有真实语义的旋钮。
- **两级降采样 + raw 受控例外**:采纳。

## 决议

1. **查询级**:query_range 的 `step` 参数,按时间窗口自适应(如 6 小时窗口 step=30s,返回约 720 点/序列)
2. **渲染级**:Worker 内 LTTB,目标点数按图表像素宽度计算
3. **raw 模式**:不经任何降采样,仅 million-points 页使用,作为渲染能力的受控展示窗口

降采样是默认,raw 是显式声明的例外。查询指令的消息协议中包含降采样参数,定义见 `docs/data-contract.md`。

## 后果

- 用户在 dashboard 上 zoom 至原始精度时,需对可视窗口重新发起小范围 raw 查询(增量拉取,而非全量)
- LTTB 只允许在 Worker 内执行(CLAUDE.md 硬约束 3),主线程永远只消费已压缩的列式快照
- million-points 页成为 MCP 渲染验证(ADR-0007)的标的
